# Codex Review Instructions

## 目的

このリポジトリでは、Codex に Pull Request レビューを依頼する際、スタイル指摘だけに偏らず、要件充足・デグレ・周辺影響まで確認する。

## 言語

- レビューコメント、総評、補足はすべて日本語で記述する。

## 参照順序

Pull Request または差分をレビューするときは、次の順で確認する。

1. PR本文
2. 関連Issue
3. `spec.md`
4. 変更差分
5. テスト差分
6. `Docs/review/README.md`
7. `Docs/review/functional-review-checklist.md`
8. `Docs/review/non-functional-and-regression-checklist.md`

PR本文が簡素でも、差分・関連Issue・`spec.md` から変更意図を再構成してレビューする。
未記入そのものを主な指摘にはしない。
それでも判断できない部分だけを「前提不足」として明記する。

`Docs/review/` 配下のファイルは詳細基準であり、ルート `AGENTS.md` から明示参照してレビュー時に読む前提とする。
`Docs/review/` だけを置いても、自動的にレビュー指示として扱われるとは限らない。

## レビュー優先順位

1. 変更内容が機能要件を満たしているか
2. 既存機能へのデグレや他機能への影響がないか
3. 非機能要件の重大な懸念がないか
4. 保守性・安全性に実害のある実装上の問題がないか
5. コーディング規約や作法の指摘は、実害がある場合だけ行う

## 必須レビュー観点

- 変更機能の期待動作と受け入れ条件が満たされているか
- 想定ユースケースだけでなく、代表的な境界条件や失敗経路が考慮されているか
- 既存の保存データ、再生、書き出し、モバイル操作、エラーハンドリングに悪影響がないか
- 変更差分に対して、妥当なテストまたは検証方法が用意されているか
- 性能、信頼性、セキュリティ、保守性の観点で重大な後退がないか
- 仕様が曖昧な場合は、想定している要求を仮説として整理し、追加で詰めるべき点を示す

## 前提の検証ルール

- 値やデータが必須か任意かは、PR本文・Issue・`spec.md` だけでなく、型定義、スキーマ、保存/読込コード、テストでも確認する
- 単独の防御コード（`??`, optional chaining, null guard など）だけを根拠に、現行契約が optional になったと断定しない
- 現行コードベース上で到達可能か確認できないケースは、断定指摘ではなく「もしその入力を許容する設計なら」と前提付きで書く
- 前提が未確定の指摘は、重要度を抑えるか open questions / assumptions に回す

## 指摘の書き方

- 指摘は重要度順に並べる
- 各指摘には、少なくとも次の4点を含める
  - 問題点
  - 背景・理由
  - 想定リスクまたは影響する要件・品質
  - 改善提案
- 単なる好みの差や軽微なスタイル論争は避ける
- 問題がない場合も「指摘なし」と明記し、残るテストギャップや未確認事項があれば補足する
- 仕様が固まり切っていない変更では、単に不足を責めず、レビュー観点として必要な仕様整理も提案する

## 出力方針

- まず findings を出し、その後に open questions / assumptions、最後に短い総評を置く
- 差分から確認できない事項は、その旨を明記する
- ファイルや行を特定できる場合は具体的に示す
- findings が 1 件だけでも、要件充足・デグレ・非機能の主要観点を確認した結果、他に重大指摘がないのか、未確認なのかを短く添える
