# Issue Best Practices

## 1. タイトル

- 先頭に種別を付与する: `[バグ]` `[改善]` `[ドキュメント]` `[メンテ]`
- 問題と対象を1行で表現する
- 曖昧な語を避ける（「調整」「対応」だけにしない）

## 2. 本文の最小要件

- 背景または現象
- 目的
- 具体手順またはタスク
- 受け入れ条件
- 検証方法

## 3. 種別ごとの重点

### バグ

- 再現手順は番号付きで書く
- 期待挙動と実挙動を分離する
- 環境情報（OS/ブラウザ/バージョン）を記載する

### 改善

- ユーザー価値を先に示す
- 非機能要件（性能、保守性、可用性）への影響を明示する
- スコープ外を明記して膨張を防ぐ

### ドキュメント

- どの読者向けかを明確化する
- 更新対象ファイルを具体パスで書く
- 再現性のある手順を受け入れ条件に含める

### メンテ

- リスクと緩和策を必ず記載する
- 影響範囲（ビルド、CI、依存、既存機能）を列挙する
- 検証計画を手順化する

## 4. ラベル運用

- 種別ラベルは必須: `bug` `enhancement` `documentation`
- 優先度ラベルは基準を固定する（例: `priority:high`）
- 1Issue1目的を守り、複数目的を1つに混在させない

## 5. 完了判定

- 受け入れ条件の全チェック完了
- 検証結果をコメントで記録
- 関連PR/コミット/ドキュメントへの参照を残す
