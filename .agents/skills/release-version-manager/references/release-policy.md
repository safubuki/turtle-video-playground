# リリース更新ポリシー

## 目的

- バージョン更新を定型化する
- `version.json` の履歴を軽量に保つ
- Git 操作のうち危険度の高いものは確認付きにする

## 基本方針

1. `version.json` には現在バージョンと最新差分だけを保持する
2. 次バージョンは AI が候補を出し、最終決定はユーザーが行う
3. コミットログが曖昧な場合は、変更ファイルと diff を優先して読む
4. `git push` は必ず別確認にする

## 推奨フェーズ

### Phase 1: 情報収集

- 最新タグ取得
- コミット取得
- 変更ファイル取得
- `version.json` 現在値取得

### Phase 2: AI要約

- 次バージョン候補
- summary
- highlights

### Phase 3: ファイル更新

- `version.json` 更新
- テスト / ビルド

### Phase 4: Git 操作

- コミット
- ローカルタグ
- リモート push

## 承認ゲート

- version 候補確定前
- `version.json` 書き込み前
- ローカルタグ作成前
- リモート push 前

## 非推奨

- 全履歴を `version.json` に保持する
- コミットログだけで履歴要約を決める
- 検証前にタグを振る
- 明示確認なしで remote push する
