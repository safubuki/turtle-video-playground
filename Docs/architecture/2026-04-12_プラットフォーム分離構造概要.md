# プラットフォーム分離構造概要

## 目的

本書は、2026-04-12 時点で完了した `standard` / `apple-safari` 分離対応について、変更前後で構造がどう変わったか、どこまで分離できたか、何を共有として残しているかを整理するための要約です。

結論を先に書くと、今回の対応で **runtime ownership は分離完了** です。
一方で、**アプリ全体を二重実装したわけではありません**。保存スキーマ、純粋 utility、共通 UI、export quality を担う一部戦略は shared のまま残しています。つまり「100% コード複製による完全分離」ではなく、**責務境界を固定した上での完全分離** です。

## 変更前の構造

変更前は、Android / PC と iPhone / iPad Safari の差分を、shared 側の実装に直接抱え込む構造でした。

| 観点 | 変更前 |
| --- | --- |
| App 入口 | 入口で flavor を一度だけ決める構造ではなく、下位レイヤーで platform 条件が増えやすい |
| Preview | `TurtleVideo.tsx` と shared preview 周辺に Safari 配慮を積み増しやすい |
| Audio | AudioContext の workaround が shared 側へ混ざりやすい |
| Export | Safari の MediaRecorder 系と standard の WebCodecs 系が shared hook に近い位置で混在しやすい |
| Save / Load | UI 側で保存導線や capability 分岐が重複しやすい |
| UI / Help | iPhone 向け案内と標準案内が複数箇所で分散しやすい |
| Test | helper 単位や capability 単位の確認が中心で、flavor シナリオ回帰の固定が弱い |

この構造では、iOS Safari の安定化修正が shared へ入り込みやすく、結果として Android / PC 側も同じ修正の影響を受けやすい状態でした。

## 変更後の構造

変更後は、入口で flavor を一度だけ解決し、その flavor が preview / export / save runtime を注入する構造へ移しています。

| 観点 | 変更後 |
| --- | --- |
| App 入口 | `src/App.tsx` が `resolveAppFlavor()` を 1 回だけ評価し、`StandardApp` または `AppleSafariApp` を lazy load する |
| Preview | `previewRuntime` 境界を介して flavor-owned preview modules を注入する |
| Audio | Safari 専用 audio workaround は `src/flavors/apple-safari/preview/` に閉じる |
| Export | `exportRuntime` 境界を介して strategy order と hook ownership を flavor 側へ寄せる |
| Save / Load | `saveRuntime` と `projectPersistence` adapter 境界で flavor 差し替え可能にする |
| UI / Help | `appFlavor` と `appFlavorUi` を通じて flavor-aware copy を出し分ける |
| Test | standard / apple-safari の flavor 回帰テストと schema 互換テストで境界を固定する |

## 根拠

### 1. App 入口で flavor を一度だけ解決している

- `src/App.tsx` は `resolveAppFlavor()` の結果に応じて `StandardApp` と `AppleSafariApp` を切り替える
- `src/flavors/standard/StandardApp.tsx` は `appFlavor="standard"` と standard runtime 群を `TurtleVideo` に注入する
- `src/flavors/apple-safari/AppleSafariApp.tsx` は `appFlavor="apple-safari"` と apple-safari runtime 群を `TurtleVideo` に注入する

この時点で、どちらの系統で動くかは App 入口で固定され、shared UI は「選ばれた runtime を使う側」に下がっています。

### 2. Preview は flavor-owned runtime へ分離されている

- `src/components/turtle-video/previewRuntime.ts` は preview の契約だけを定義する
- `src/flavors/standard/standardPreviewRuntime.ts` は standard 用 preview hook / policy / capability を束ねる
- `src/flavors/apple-safari/appleSafariPreviewRuntime.ts` は apple-safari 用 preview hook / policy / capability を束ねる

特に capability 正規化が分かれており、standard 側は `isIosSafari: false` を強制し、apple-safari 側は `isIosSafari: true` を強制します。これにより、「同じ shared component が実行時に Safari 判定を増やして振る舞いを変える」のではなく、「最初から別 runtime を使う」構造になっています。

### 3. Export は strategy ownership を flavor 側へ移している

- `src/components/turtle-video/exportRuntime.ts` は export hook 契約だけを持つ
- `src/flavors/standard/standardExportRuntime.ts` は standard 系 export hook を公開する
- `src/flavors/apple-safari/appleSafariExportRuntime.ts` は apple-safari 系 export hook を公開する

Safari 専用 MediaRecorder 経路は `src/flavors/apple-safari/export/` が所有し、standard 側は WebCodecs 優先の系統を維持します。shared 側には export contract と共通 core だけを残し、platform 特有の分岐を抱え込まない構造に変わりました。

### 4. Save / Load は runtime 境界と shared schema を分離した

- `src/components/turtle-video/saveRuntime.ts` は save runtime の契約だけを定義する
- `src/flavors/standard/standardSaveRuntime.ts` と `src/flavors/apple-safari/appleSafariSaveRuntime.ts` は flavor ごとの configure 処理を持つ
- `src/stores/projectPersistence.ts` は shared schema を維持したまま persistence adapter を注入可能にする

現時点では standard も apple-safari も IndexedDB adapter を使っていますが、これは「まだ同じ保存実装で足りる」ため shared のままにしているだけです。境界そのものはすでに確保されているため、将来どちらか片方だけ保存要件が変わっても UI を壊さず差し替えられます。

### 5. UI と Help は flavor-aware だが platform 直判定を持たない

- `src/app/appFlavorUi.ts` に flavor badge / support summary / download guidance / preview notice / save guidance を集約した
- `src/constants/sectionHelp.ts` は flavor 文脈を受ける生成関数へ移行した
- `Header.tsx`、`PreviewSection.tsx`、`SaveLoadModal.tsx`、`SectionHelpModal.tsx` は `appFlavor` と capability を受けて描画する

これにより、shared UI は「iPhone なら if 分岐する」実装ではなく、「注入された flavor の説明を表示する」実装へ変わっています。

### 6. テストが flavor 単位で回帰を固定している

- `src/test/standardFlavorRegression.test.ts` は standard line が Safari warm-up や Safari audio recovery を前提にしないことを固定する
- `src/test/appleSafariFlavorRegression.test.ts` は apple-safari line の video → image → video、BGM mixed routing、visibility 復帰、seek 復帰、MediaRecorder 優先 export を固定する
- `src/test/stores/projectStoreSave.test.ts` は standard save → apple-safari load の round-trip と legacy narration 互換を固定する
- 既存の isolation / capability tests と合わせて、runtime 境界と shared schema 互換の両方を監視する

Phase 7 完了時点では `npm run quality:gate` が通過し、38 test files / 294 tests pass を確認しています。

## 完全分離されたのか

### 判定

**はい。ただし意味は「責務境界として完全分離」です。**

本対応で完了したのは、次の意味での完全分離です。

- App 入口で flavor が一度だけ決まる
- preview / export / save の active runtime 所有権が flavor ごとに分かれている
- iPhone / iPad Safari の workaround を standard runtime が持たない
- standard 側の改善で apple-safari runtime 実装を直接触らなくてよい
- apple-safari 側の安定化修正で standard runtime 実装を直接触らなくてよい
- shared schema と shared helper の変更はテストで両系統へ波及確認できる

### 100% 二重実装ではない理由

次は shared のまま残しています。これは未分離ではなく、**意図的な共通契約** です。

- project save schema
- `projectPersistence` adapter 境界
- 純粋な timeline / format utility
- `AppShell` や一部 UI 部品
- export quality のための shared pre-render strategy

したがって、本件は「全部のコードを複製して完全に別アプリ化した」のではなく、**共有すべき契約は shared に残し、runtime 依存の挙動だけを完全分離した** と表現するのが正確です。

## 変更前後の見取り図

### 変更前

```text
App / TurtleVideo
  ├─ shared preview / audio / export / save
  ├─ platform 条件分岐
  └─ iOS Safari workaround が shared に混ざりやすい
```

### 変更後

```text
App.tsx
  └─ resolveAppFlavor()
      ├─ StandardApp
      │   └─ TurtleVideo(appFlavor=standard, previewRuntime, exportRuntime, saveRuntime)
      └─ AppleSafariApp
          └─ TurtleVideo(appFlavor=apple-safari, previewRuntime, exportRuntime, saveRuntime)

shared
  ├─ UI shell
  ├─ project schema
  ├─ common utilities
  └─ runtime contracts

flavor-owned
  ├─ src/flavors/standard/**
  └─ src/flavors/apple-safari/**
```

## 今後の依頼の出し方

今後は「どの platform 向けか」だけでなく、**どの境界を触る依頼か** を一緒に指定すると、より正確に対応できます。

### 推奨する指定方法

1. platform を明示する
2. 影響範囲を明示する
3. shared を触ってよいかを明示する
4. 必要なら回帰テスト対象を指定する

### 依頼文の例

- Android / PC 向けだけを直したい場合
  - 「standard flavor の preview だけ修正して」
  - 「Android / PC 向けの export 改善。apple-safari line は触らないで」
- iPhone / iPad Safari 向けだけを直したい場合
  - 「apple-safari flavor の preview 安定化をして」
  - 「iPhone Safari 向けの save / load 導線だけ改善して。shared schema は変えないで」
- 共通契約も変える場合
  - 「standard / apple-safari 共通で project schema を拡張して。save round-trip test も更新して」
  - 「shared export pre-render strategy を変更して。両 flavor regression test を確認して」

### 曖昧になりやすい依頼

- 「iPhone向けに対応して」だけだと、preview / export / save / UI-help のどこを触るかが曖昧です
- 「Android向けに最適化して」だけだと、standard runtime だけでよいのか、shared utility の改善も含むのかが曖昧です

### 最低限この形にすると安全

```text
{target flavor} の {preview/export/save/UI-help/shared schema} を対象に、
{何を改善するか} を対応して。
shared を触る必要がある場合だけ触って、関連テストも更新して。
```

## 実務上の判断ルール

- Android / PC は基本的に `standard` として扱う
- iPhone / iPad Safari は基本的に `apple-safari` として扱う
- 「iPhone 向け」と言われた場合でも、Safari 以外の iOS ブラウザまで含むなら追加確認が必要
- 「両方に効く改善」の場合だけ shared を触る
- shared を触ったら、flavor regression と schema compatibility の両方を確認する
