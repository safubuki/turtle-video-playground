# 機能要件レビュー観点

## 1. 変更の目的を固定する

レビュー開始時に、次を短く整理する。

- この PR は何を変えるのか
- 誰のどの操作が変わるのか
- 完了条件は何か
- 対象外は何か

これが PR 本文に明示されていなくても、関連Issue、`spec.md`、差分から暫定的に再構成する。
どうしても判断できない部分だけを前提不足として扱う。

## 2. 要件充足の確認

次を確認する。

- 実装が PR 本文や Issue の目的に対応しているか
- 必要な処理が差分内で一通りそろっているか
- UI 変更なら、表示だけでなく保存・再表示・操作結果までつながっているか
- データ変更なら、作成・更新・読込・後方互換まで考慮されているか
- 失敗時や未設定時のフォールバックがあるか
- 値やデータの必須/任意が論点になる場合、型・スキーマ・保存/読込コード・テストに照らして現行契約を確認したか
- `??`, optional chaining, null guard など単独の防御コードだけで、optional な仕様だと早合点していないか

## 3. 受け入れ条件の確認

最低限、次の観点で確認する。

- 正常系が成立するか
- 境界値で破綻しないか
- 未入力・空データ・無効値で壊れないか
- 既存データや既存設定のままでも動くか
- テストや確認手順が、変更の本質に届いているか
- 想定している失敗経路や空データ経路が、現行契約上で本当に到達可能か
- 到達可能性が未確認なら、断定指摘ではなく前提付きの open question として扱っているか

## 4. Turtle Video で見落としやすい代表機能

このプロジェクトでは、変更箇所が直接そこに触れていなくても、次への影響を意識する。

- タイムライン編集
- プレビュー再生
- エクスポート
- 自動保存 / 手動保存 / 復元
- BGM / ナレーション / キャプション
- モバイル操作

## 5. 指摘に値する典型例

- UI だけ追加され、保存や復元に反映されていない
- 正常系は動くが、既存データ互換がない
- 受け入れ条件にある操作が、別導線では成立しない
- テストが実装詳細だけを見ており、要件の成立を担保していない
- PR の説明と実装内容がズレている
- 作成者は「困りごと」だけ書いているのに、レビュー側が要件仮説を立てずに形式不足だけを指摘している
- 防御コード 1 か所だけを見て、型やテストが示す現行契約と矛盾する仮説を高優先度指摘にしている
