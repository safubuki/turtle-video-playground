# AIレビュー運用ガイド

このフォルダは、Codex に PR レビューを依頼するときの基準をまとめたものです。

## 目的

- レビューを日本語で統一する
- 文法・作法だけでなく、機能要件の充足を確認する
- 非機能要件は重くし過ぎず、重大リスクだけ拾う
- デグレや他機能影響を見落としにくくする
- 指摘時に背景と改善案まで含める

## この構成の考え方

- ルート `AGENTS.md` は Codex 向けの入口指示とする
- `Docs/review/` は、その入口から読ませる詳細基準とする
- つまり `AGENTS.md` に「何を読むか」を書き、`Docs/review/` に「どうレビューするか」の詳細を書く
- `Docs/review/` だけを置いても、Codex が常に自動参照する前提ではない

## 運用方針

- PRテンプレートは軽く保ち、作成者には「何を変えたか」「なぜ変えたか」「何が不安か」を中心に書いてもらう
- レビュー側は、足りない観点を差分・Issue・`spec.md` から補完する
- 仕様が曖昧な変更では、欠点の指摘だけでなく、詰めるべき仕様も提案する

## ドキュメント構成

- `Docs/review/functional-review-checklist.md`
  - 機能要件と変更仕様の充足を確認するための観点
- `Docs/review/non-functional-and-regression-checklist.md`
  - 非機能要件とデグレ観点を軽量に確認するための観点

## レビュー時の前提

レビューでは、まず次の情報を確認する。

1. PR本文
2. 関連Issue
3. `spec.md`
4. 変更差分
5. テスト差分
6. 型定義・スキーマ・保存/読込コード・既存テストから読み取れる現行契約

情報が薄い場合は、まず差分や関連情報から意図を再構成する。
それでも判断できない部分だけを「前提不足」として扱う。

## 前提検証ルール

- 値やデータの必須/任意は、PR本文や差分の見た目だけでなく、型・スキーマ・読込/保存・テストで確認する
- `??`, optional chaining, null guard などの防御コードは、実装の安全策である可能性がある。単独では仕様変更の根拠にしない
- 現行契約上で到達可能か確認できないケースは、断定せず仮説として扱う
- 仮説ベースの指摘は「もしこのケースを許容する設計なら」と前提を明記する

## レビューコメントの期待フォーマット

各指摘は、次の形を基本とする。

1. 問題点
2. なぜ問題か
3. どの要件・品質・既存機能に影響するか
4. どう直すとよいか

短くてよいが、単なる NG 指摘で終わらせない。

## レビュー完了時の最低ライン

- findings がある場合も、要件充足・デグレ・非機能の主要観点を確認した結果、他に重大指摘がないのか、未確認なのかを添える
- findings がない場合は「指摘なし」と書いた上で、未確認事項やテストギャップだけを補足する

## 指摘しなくてよいもの

- 個人の好みに近い命名差
- 現時点で実害が読み取れない細かな書き方の違い
- 既存コードベース全体で許容されている軽微な揺れ

## 指摘を優先するもの

- 要件未達
- 既存動作の破壊
- 保存データ互換性の欠落
- 例外系・失敗系の抜け
- テスト不足に起因する回帰リスク
- 性能や信頼性の明確な後退
