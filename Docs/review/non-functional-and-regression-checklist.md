# 非機能・デグレレビュー観点

非機能レビューは重くし過ぎず、差分に関係する重大リスクを中心に見る。

## 1. 非機能の軽量チェック

### 性能

- 不要な再描画や高頻度処理を増やしていないか
- メディア処理や保存処理で、明らかな待ち時間増加がないか

### 信頼性

- 例外時に中途半端な状態で止まらないか
- 非同期処理の競合や後始末漏れがないか
- 問題として挙げる失敗経路が、現行契約上で実際に到達し得るか

### セキュリティ / プライバシー

- API キー、URL、ユーザー入力の扱いが不用意に露出していないか
- 外部通信や保存データの扱いに新しい漏えい経路がないか

### 保守性

- 変更理由がコードから追えない複雑な分岐を増やしていないか
- 既存パターンを壊す独自実装を持ち込んでいないか

## 2. デグレ確認

次の観点で既存機能への影響を確認する。

- 共有ストアや共通ユーティリティを変えた場合、別機能にも影響しないか
- 既存の保存データ、復元データと互換性があるか
- 既存テストの想定を壊していないか
- 変更差分の近接領域に、別機能の前提が埋め込まれていないか
- 既存テストやスキーマが保証している前提を無視して、仮説だけでデグレ扱いしていないか

## 3. Turtle Video の重点チェック

このプロジェクトでは、特に次の退行に注意する。

- `URL.createObjectURL` を増やしたのに解放がない
- メディア要素の表示制御を `display: none` にしてしまう
- AudioContext / MediaElementSource の生成条件を壊す
- 書き出し時の黒画面、静止画化、音ズレを再発させる
- モバイルのスライダー誤操作防止を崩す
- IndexedDB 保存や自動保存の後方互換を壊す

## 4. 指摘の深さ

- 重大リスクがなければ、非機能指摘は短くまとめる
- 実害の薄い細部最適化は、レビューの主題にしない
- ただし、今後の障害につながりやすい共通基盤の変更は優先して拾う
- 到達性や前提が未確認の懸念は、断定口調や高い重要度にしない
- ローカルな 1 指摘だけで終わらせず、他の重大な非機能・デグレ懸念がなかったかも短く添える
